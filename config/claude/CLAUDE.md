
## Working Relationship

- If my approach is wrong, say so directly. Challenge flawed reasoning, but don't disagree for the sake of it.
- Don't provide timeline estimates or predictions for tasks.
- Flag uncertainty explicitly: distinguish "confident" vs "best guess, thin evidence" vs "speculating."
- When there's a tradeoff, present options with evidence. Don't silently pick the easy path.
- When reviewing code, actively look for problems rather than validating the approach.

## Thinking Economy

Minimize internal reasoning verbosity:

- Per-thought limit: 10 words
- Use abbreviated notation: "Spec->X; File->Y; Apply Z"
- DO NOT narrate phases ("Now I will verify...")
- Execute tasks silently; output results only

Examples:

- VERBOSE: "Now I need to check if the imports are correct. Let me verify..."
- CONCISE: "Imports: check stdlib, add missing"

## Plan Mode

- Make the plan extremely concise. Sacrifice grammar for the sake of concision.
- At the end of each plan, give me a list of unresolved questions to answer, if any.

## Commiting

- Don't add yourself as the author of the commit. Leave it blank so that I can add myself as the author when I commit.

## Attribution

- Never claim credit or responsibility for any work. No "Co-Authored-By", "written by Claude", "generated by AI", or similar attribution in commits, PR descriptions, code comments, or anywhere else.

## ESLint

- Never disable ESLint rules with `eslint-disable` comments as a quick fix. Instead, research the rule's intent and find a correct solution (fix the code, adjust config with justification, or consult docs).
- If a rule genuinely needs to be disabled (e.g., false positive, intentional pattern), prompt the user first with reasoning and documentation before adding any `eslint-disable` comment.

## .npmrc auth tokens

Never use `${ENV_VAR}` substitution in a committed `.npmrc` for auth tokens. pnpm warns on every invocation when the var is unset — including non-install commands like `typecheck` and `lint` — and env propagation through turbo/monorepo child processes is unreliable. Instead: have CI append the auth line at runtime (e.g. `echo '//registry/:_authToken=...' >> .npmrc`); have local devs configure once via `pnpm config set //registry/:_authToken <token>` (writes to `~/.npmrc`).

## wt worktrees

When `.wt-meta` is present in the repo root, this is a worktree managed by the `wt` CLI. For branch completion and cleanup, use the `wt` commands — don't do manual git operations:

- `wt finish` — rebase onto base branch, fast-forward, remove worktree + branch
- `wt drop` — abandon worktree without merging, remove worktree + branch
- `wt sync` — rebase onto latest base branch from origin

These handle worktree-specific edge cases (base branch checked out elsewhere, CWD cleanup) that manual git commands will break on.

## TypeScript

- Never add ambient type declarations (`declare module "*.png"`, etc.) that the framework already provides (e.g., Next.js via `next-env.d.ts`). Duplicate ambient declarations cause non-deterministic errors across OS filesystems (macOS vs Linux process files in different order).
- In Next.js projects, always commit `next-env.d.ts` and remove it from `.gitignore`. It is generated by `next dev` locally but missing on CI when `tsc` runs standalone, causing TS2307 errors. Also add `**/next-env.d.ts` to the root ESLint config's global ignores — it uses triple-slash refs intentionally and must not be linted.
